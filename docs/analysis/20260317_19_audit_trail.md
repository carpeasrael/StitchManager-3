# Analysis: Audit Trail — Change History Logging

**Issue:** GitHub #100
**Date:** 2026-03-17

---

## 1. Problem Description

Per project.md section 9.1: "Aenderungen an Kalkulationen, Materialien, Lizenzdaten und Projektstatus muessen nachvollziehbar dokumentiert werden." Currently only `updated_at` timestamps exist — no record of what changed, who changed it, or what the previous value was. Acceptance criterion 10 requires full project history traceability.

## 2. Affected Components

| File | Impact |
|------|--------|
| `src-tauri/src/db/migrations.rs` | Migration v21: `audit_log` table |

| `src/services/ReportService.ts` | `getAuditLog` function |
| `src/components/ManufacturingDialog.ts` | History view in detail panes |

## 3. Root Cause

No audit infrastructure was built during any sprint. All update commands modify data in-place with only `updated_at` timestamp.

## 4. Proposed Approach

### Step 1: Migration v21 — `audit_log` table
```sql
CREATE TABLE audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_type TEXT NOT NULL,  -- 'project', 'material', 'license', 'order'
    entity_id INTEGER NOT NULL,
    field_name TEXT NOT NULL,
    old_value TEXT,
    new_value TEXT,
    changed_by TEXT,
    changed_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Step 2: Rust helper function
A reusable `log_field_change(conn, entity_type, entity_id, field, old, new)` helper.

### Step 3: Instrument update commands
For each update command, query old values before UPDATE, then compare with new values and log differences.

### Step 4: Query command
`get_audit_log(entity_type, entity_id)` → `Vec<AuditLogEntry>`

### Step 5: Frontend — history display in detail panes
A collapsible "Aenderungshistorie" section showing audit entries per entity.
