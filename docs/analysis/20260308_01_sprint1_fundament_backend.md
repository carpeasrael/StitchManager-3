## 1. Problem Description

Sprint 1 establishes the backend foundation for the StitchManager v2 Tauri application. The current codebase contains only a minimal scaffold: `lib.rs` with a basic Tauri builder (sql + log plugins), a `Cargo.toml` with 5 dependencies, and a `tauri.conf.json` with default window settings. There is no module structure, no error handling, no database schema, and only 2 files under `src-tauri/src/`.

Sprint 1 must deliver:

1. **Rust module structure** — organized directories for db, commands, parsers, services, and error handling
2. **Central error type** (`AppError`) with `thiserror` derivation and `serde::Serialize` for Tauri IPC
3. **12 new Cargo dependencies** for the full backend feature set
4. **SQLite schema** — 10 tables with migration logic, Rust model structs, and default settings
5. **Tauri window configuration** — corrected title, minimum window size, DB initialization in setup hook

After Sprint 1, the app must start, create the database with schema version 1, and have a clean module structure ready for Sprint 2 (Ordner & Dateien).

## 3. Root Cause / Rationale

### Why these changes are needed

The current scaffold was generated by `npm create tauri-app` and contains only the bare minimum to compile. Before any feature work (folder management, file parsing, AI integration) can begin, the project needs:

- **Module structure**: Rust demands explicit `mod` declarations. Without `db/`, `commands/`, `parsers/`, `services/`, `error.rs`, all subsequent sprint work would need to restructure the crate.
- **AppError**: Tauri commands must return `Result<T, E>` where `E: serde::Serialize`. Without a central error type, every command would need ad-hoc error handling. The `thiserror` crate provides `#[from]` for automatic conversion from `rusqlite::Error`, `io::Error`, etc.
- **Dependencies**: The 12 new crates cover all backend capabilities needed across all sprints (database, async runtime, HTTP client, file watching, image processing, directory scanning, date/time, hashing, base64 encoding, UUIDs, error handling, binary parsing). Adding them now avoids repeated `Cargo.toml` edits later.
- **Database schema**: The schema is the backbone of the application. All 10 tables must exist before any CRUD commands can be built. The migration system (`schema_version` table) ensures the schema can evolve in future releases.
- **Window configuration**: The title "StichMan" is the product name per the proposal. Minimum window size (960x640) prevents layout breakage in the 3-panel CSS grid.

### Why rusqlite alongside tauri-plugin-sql

Per proposal section 2.3, a **dual strategy** is used:
- **Backend (rusqlite)**: Write operations, migrations, complex queries, batch operations, scanner results — all validated and transactional
- **Frontend (tauri-plugin-sql)**: Lightweight read queries for UI updates, settings access

The backend owns schema migration and data integrity; the frontend gets fast read access without IPC round-trips for simple queries.

### Ticket S1-T2: AppError-Typ und Serialisierung

**Goal**: Implement the central `AppError` enum per proposal section 4.7.

**Steps**:

1. In `src-tauri/src/error.rs`, implement:
   ```rust
   #[derive(Debug, thiserror::Error)]
   pub enum AppError {
       #[error("Datenbankfehler: {0}")]
       Database(#[from] rusqlite::Error),

       #[error("Dateifehler: {0}")]
       Io(#[from] std::io::Error),

       #[error("Parserfehler: {format} -- {message}")]
       Parse { format: String, message: String },

       #[error("KI-Fehler: {0}")]
       Ai(String),

       #[error("Nicht gefunden: {0}")]
       NotFound(String),

       #[error("Validierungsfehler: {0}")]
       Validation(String),
   }
   ```

2. Implement `serde::Serialize` manually (not via derive, because `rusqlite::Error` and `io::Error` are not Serialize):
   ```rust
   impl serde::Serialize for AppError {
       fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
       where S: serde::Serializer {
           serializer.serialize_str(&self.to_string())
       }
   }
   ```

3. Add `pub use error::AppError;` in `lib.rs` or ensure it is accessible from `commands/` modules via `crate::error::AppError`.

**Verification**: `cargo check` passes. The `AppError` type satisfies `Debug + thiserror::Error + serde::Serialize`, which is required for Tauri command return types.

**Dependencies**: Requires S1-T1 (file structure) and S1-T3 (`thiserror` and `rusqlite` crates).

### Ticket S1-T4: SQLite-Schema implementieren (10 Tabellen)

**Goal**: Implement the full database schema from proposal section 3, with migration logic and Rust model structs.

**Steps**:

#### Step 1: Implement `db/migrations.rs`

Create two public functions:

- `pub fn init_database(db_path: &str) -> Result<rusqlite::Connection, AppError>` — opens/creates the SQLite file, calls `run_migrations`
- `pub fn run_migrations(conn: &rusqlite::Connection) -> Result<(), AppError>` — checks `schema_version`, applies missing migrations

The initial migration (version 1) creates all 10 tables + junction tables:

1. **schema_version** — migration tracking (section 3.1)
2. **folders** — managed folder entries with parent_id for hierarchy (section 3.2), indexes on `parent_id` and `path`
3. **embroidery_files** — main file table with 18 columns (section 3.3), indexes on `folder_id`, `name`, `filepath`, `ai_analyzed`
4. **file_formats** — format variants per file (section 3.4), indexes on `file_id` and `format`
5. **file_thread_colors** — thread color info per file (section 3.5), index on `file_id`
6. **tags** — unique tag names (section 3.6)
7. **file_tags** — n:m junction table with composite PK (section 3.6), indexes on both FKs
8. **ai_analysis_results** — AI analysis audit log (section 3.7), index on `file_id`
9. **settings** — key-value store (section 3.8)
10. **custom_field_definitions** — user-defined metadata fields (section 3.9)
11. **custom_field_values** — junction table for custom field values (section 3.9), index on `file_id`

After creating tables, insert default settings (section 3.8):
- `library_root` = `~/Stickdateien`
- `metadata_root` = `~/Stickdateien/.stichman`
- `theme_mode` = `hell`
- `ai_provider` = `ollama`
- `ai_url` = `http://localhost:11434`
- `ai_model` = `llama3.2-vision`
- `ai_temperature` = `0.3`
- `ai_timeout_ms` = `30000`
- `rename_pattern` = `{name}_{theme}`
- `organize_pattern` = `{theme}/{name}`

Insert initial `schema_version` row: `(1, 'Initial schema')`.

All DDL and DML should be wrapped in a single transaction for atomicity.

The migration system must be **idempotent**: running `init_database` on an already-initialized database must be a no-op (check `schema_version` first).

#### Step 2: Update `lib.rs` — setup hook

Add a Tauri setup hook that initializes the database at app startup:

```rust
pub fn run() {
    let mut builder = tauri::Builder::default()
        .plugin(tauri_plugin_sql::Builder::new().build())
        .setup(|app| {
            let app_data_dir = app.path().app_data_dir()
                .expect("failed to resolve app data dir");
            std::fs::create_dir_all(&app_data_dir)
                .expect("failed to create app data dir");
            let db_path = app_data_dir.join("stitch_manager.db");
            let _conn = db::init_database(
                db_path.to_str().expect("invalid db path")
            ).expect("failed to initialize database");
            Ok(())
        });

    #[cfg(debug_assertions)]
    {
        builder = builder.plugin(tauri_plugin_log::Builder::new().build());
    }

    builder
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

The `init_database` function from S1-T4 opens or creates the SQLite database and runs migrations. The connection can be dropped after setup (each command will open its own connection, or we store it in Tauri managed state — decision deferred to Sprint 2).

**Consideration**: For Sprint 2+, the connection (or a connection pool / path) should be stored in Tauri's managed state via `app.manage()`. For Sprint 1, it is sufficient to just run the init and verify the schema is created.

#### Step 3: Verify permissions

No new permissions are needed in `capabilities/default.json` for Sprint 1. The `rusqlite` database access is backend-only and does not go through the Tauri plugin permission system. The existing `sql:default` permission covers `tauri-plugin-sql` for frontend access.

## 7. Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| `rusqlite` version 0.31 may conflict with `tauri-plugin-sql` bundled SQLite | Use `bundled` feature on rusqlite so it compiles its own SQLite. Two separate SQLite instances is acceptable — they access the same file but never concurrently in Sprint 1. |
| Database path resolution differs across platforms | Use `app.path().app_data_dir()` which is Tauri's cross-platform abstraction. |
| Large dependency tree (12 crates) increases compile time | Expected; first compile will be slow but incremental builds will be fast. Consider `cargo check` during development. |
| Module placeholders may cause "unused" warnings | Add `#[allow(unused)]` or minimal placeholder content to avoid noise during development. |
