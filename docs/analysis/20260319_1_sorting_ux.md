# Analysis: User-Friendly Sorting Improvements

**Date:** 2026-03-19
**Scope:** File sorting discoverability, sort persistence, folder reordering

---

## Problem Description

1. **File sort controls are hidden.** They are buried at the bottom of the SearchBar's expandable "Erweiterte Filter" panel (SearchBar.ts:275-325), behind 15+ filter fields. Sorting is the second most common action after search, yet requires 3 clicks and scrolling to reach. Additionally, sort is counted in `activeFilterCount` (line 29), conflating sorting with filtering in the badge.

2. **Sort preference is not persisted.** On every app reload, sort resets to the default (filename ASC). Users who consistently sort by designer or date must re-apply their preference each session.

3. **Folder sorting is static.** The `sort_order` column exists in the DB schema (migrations.rs:167) but is never updated -- all folders get sort_order=0 and sort purely alphabetically. No backend command exists to update sort_order, and no UI exists for folder reordering.

4. **Missing sort fields.** Backend supports 10 sort fields (files.rs:256-274), but only 7 are exposed in the UI. Missing: `color_count`, `file_type`, `status`.

### Frontend
- `src/components/SearchBar.ts` -- contains sort controls to extract (lines 275-325) and `activeFilterCount` (line 29)

### Backend
- `src-tauri/src/commands/folders.rs` -- needs new `update_folder_sort_orders` command
- `src-tauri/src/commands/mod.rs` -- re-export
- `src-tauri/src/lib.rs` -- register new command

---

## Root Cause / Rationale

- Sort controls were added as part of the advanced filter panel, treating sorting as a secondary/power-user feature. In practice, embroidery users sort frequently (by designer, date added, category).
- The `sort_order` column was added to the DB schema but the backend CRUD commands were never updated to expose it.
- No persistence mechanism was implemented for sort preferences despite the settings infrastructure being in place.

---

## Proposed Approach

### Phase A -- Core UX (3 changes)

1. **New `SortControl` component** (`src/components/SortControl.ts`):
   - Follows the `FilterChips` pattern: extends Component, subscribes to `searchParams`, renders compact select + direction toggle button
   - Placed in `.app-menu` toolbar between search bar and format filter chips
   - All 10 sort fields exposed with German labels
   - Persists sort to settings on change (`file_sort_field`, `file_sort_direction`)

2. **Remove sort from SearchBar**:
   - Delete sort section (lines 275-325)
   - Remove `sortField` from `activeFilterCount` (line 29) so sorting no longer inflates the filter badge
   - "Alle zurucksetzen" must NOT reset sort (sort != filter)

3. **Load persisted sort on startup** in `main.ts` `initTheme()`:
   - Read `file_sort_field`/`file_sort_direction` from settings
   - Seed into initial `searchParams`

### Phase B -- File List Sort Headers (optional)

- Clickable header row above FileList with 4-5 common sort fields
- Complements toolbar dropdown for power users

### Phase C -- Folder Reordering

1. **Backend:** New `update_folder_sort_orders` command accepting `Vec<(i64, i32)>`
2. **Frontend:** Drag-and-drop in Sidebar with keyboard alternative (Alt+Up/Down)
3. **Service:** New `FolderService.updateSortOrders()` method
